# 알림톡 악용/실수 방지 정책

PetManager 알림톡은 오너의 수동 액션으로만 발송한다. 단, 수동 액션이라도 고객 도배, 실수 클릭, 같은 상태 반복 발송은 서버에서 차단한다.

## 기본 원칙

- 전화번호 단순 `1분 1건` 제한은 쓰지 않는다.
- 정상 예약 흐름에서 서로 다른 이벤트가 빠르게 이어지는 것은 허용한다.
- 같은 예약의 같은 이벤트를 반복 발송하는 것은 차단한다.
- 예약, 고객, 반려동물과 연결되지 않은 임의 번호 알림톡은 오너 API에서 차단한다.
- 프론트 비활성화와 무관하게 서버 API가 최종 방어선이다.
- 제한으로 차단된 발송은 크레딧을 차감하지 않는다.

## 허용하는 정상 흐름

같은 고객 전화번호라도 아래처럼 실제 예약 이벤트가 다르면 연속 발송을 허용한다.

- 예약 요청 접수
- 예약 확정
- 예약 변경 완료
- 미용 시작
- 픽업 준비
- 미용 완료

여러 마리 예약도 전화번호 기준으로 막지 않는다. 반려동물별로 예약 건이 다르면 서로 다른 `appointmentId` 기준의 정상 이벤트로 본다.

## 차단하는 경우

- 이미 `미용 시작` 상태인 예약에서 미용 시작 버튼을 다시 누르는 경우
- 이미 `픽업 준비` 상태인 예약에서 픽업 준비 버튼을 다시 누르는 경우
- 이미 `완료`, `취소`, `거절`, `확정` 상태인 예약에 같은 상태 버튼을 다시 누르는 경우
- 같은 예약에 같은 알림 타입이 이미 `queued`, `sent`, `mocked` 상태로 있으면 재발송 차단
- 예약 변경 완료 알림은 날짜, 시간, 서비스, 담당자, 소요시간, 안내 시간 중 실제 변경이 없으면 차단
- 같은 예약 일정 스냅샷으로 `예약 변경 완료` 또는 `예약 시간 제안` 알림을 다시 보내는 경우
- 예약/고객/반려동물 연결 없이 임의 전화번호로 알림톡을 보내는 경우

## 중복 판정 기준

알림 발송 시 `notifications.metadata.abuseDedupeKey`를 기록한다.

- 일반 상태 이벤트: `appointment:{appointmentId}:type:{notificationType}`
- 예약 변경/시간 제안: `appointment:{appointmentId}:type:{notificationType}:snapshot:{date|time|service|staff}`
- 수동 메시지: `manual:{type}:recipient:{guardianOrPetOrPhone}:message:{messageHash}`

기존 알림에 dedupe key가 없는 경우에도, 같은 예약과 같은 알림 타입의 기존 성공/대기 알림은 중복으로 본다.

## 크레딧

- 차단된 발송은 `skipped`로 처리하고 크레딧을 예약/차감하지 않는다.
- Provider 발송 실패는 기존 정책대로 예약된 크레딧을 환불한다.
- 성공한 알림톡만 샵 크레딧 1건을 확정 차감한다.
