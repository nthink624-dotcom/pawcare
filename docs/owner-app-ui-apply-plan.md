# owner-app UI Apply Plan

## 1. owner-app.tsx 안에서 공통 컴포넌트로 바꿀 수 있는 UI 블록 목록

- 상단 섹션 헤더
  - 화면 제목
  - 보조 설명
  - 우측 보조 액션
- 홈 요약 카드
  - 승인 대기
  - 예약 현황
  - 완료 내역
  - 취소·변경
- 공통 리스트 row
  - 고객 리스트 row
  - 예약 리스트 row
  - 설정 메뉴 row
- 상태 배지
  - 예약 상태
  - 고객 알림 상태
  - 운영 상태
- 빈 상태 블록
  - 예약 없음
  - 고객 없음
  - 오늘 처리할 항목 없음
- 관리 바
  - 검색 바
  - 선택 모드 바
  - 필터/액션 바

## 2. 먼저 바꿀 것

1. `SectionHeader`
   - 반복되는 섹션 제목 블록 정리
2. `StatusBadge`
   - 예약/고객 상태 문법 통일
3. `EmptyState`
   - 예약 없음, 고객 없음, 오늘 항목 없음 화면 통일
4. `ListRow`
   - 고객/설정 리스트의 기본 row 구조 정리
5. `AppButton`
   - 화면 하단 메인 CTA와 카드 안 소형 액션의 스타일 기준 맞추기

이 단계에서는 시각 블록만 치환하고, 상태 분기나 데이터 계산은 그대로 둔다.

## 3. 나중에 바꿀 것

- 홈 상단 프로필-매장명 헤더 블록
- 예약 상세/변경 시트
- 고객 상세/편집 시트
- 방문 캘린더 팝업
- 미용 기록 편집 폼
- 각 탭별 큰 컨테이너 분리

이 영역은 현재 UI와 상태 로직이 밀접하게 얽혀 있으므로 1차 공통 컴포넌트 적용 후 별도 단계로 분리한다.

## 4. 절대 건드리면 안 되는 로직

- 예약 승인/거절/완료/취소/변경 로직
- 고객 추가/수정/삭제/복구 로직
- 알림톡 on/off 및 발송 분기 로직
- 매장 전환 로직
- 서비스/영업시간/가용시간 계산
- 결제/플랜/구독 상태 계산
- 관리자 세션/오너 세션 판단 로직
- API 호출, validation, routing, Supabase 저장 로직

## 5. 예상 리스크

- `owner-app.tsx`는 홈, 예약 조회, 고객관리, 설정 진입이 한 파일에 섞여 있어 작은 JSX 치환도 이벤트 핸들러 연결 누락으로 이어질 수 있다.
- 상태 배지를 공통화할 때 현재 내부 상태 문자열과 표시 문구 간 매핑이 깨질 수 있다.
- 리스트 row를 공통화할 때 체크 모드와 일반 모드의 클릭 영역이 달라질 수 있다.
- 바텀시트/팝업 영역은 시각적으로는 분리 가능해 보여도 실제로는 상태 전환이 강하게 얽혀 있다.

## 6. 적용 순서

1. `SectionHeader`, `StatusBadge`, `EmptyState`를 먼저 도입한다.
2. 홈 화면의 순수 표시 블록부터 교체한다.
3. 고객관리에서 읽기 전용 row부터 `ListRow`로 바꾼다.
4. 예약조회에서 읽기 전용 row와 상태 배지만 공통화한다.
5. 설정 진입 row를 `ListRow`로 맞춘다.
6. 그 다음에만 `AppButton`의 사용처를 단계적으로 맞춘다.
7. 바텀시트, 폼, 캘린더, 상세 편집 레이어는 별도 단계에서 다룬다.

핵심 원칙:
- 먼저 JSX 시각 블록만 통일한다.
- 상태와 핸들러는 기존 위치에 유지한다.
- 한 번에 큰 구조를 갈아엎지 않는다.
