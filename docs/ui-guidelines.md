# UI Guidelines

## 1. 전체 디자인 원칙

- PetManager는 하나의 UI 문법만 쓰는 서비스가 아니다.
- PetManager의 사용자-facing 화면은 화면군별 문법을 분리한다.
- 오너 운영 화면은 모바일 앱 문법을 따른다.
- 소비자 예약 화면은 모바일 단계형 플로우 문법을 따른다.
- 관리자 페이지는 PC 웹 콘솔 문법을 따른다.
- 향후 고객관리 PC 버전은 모바일 고객관리 화면을 단순 확대하지 않고 별도 화면군으로 설계한다.
- 같은 기능이라도 모바일과 PC에서 레이아웃 문법이 다를 수 있다.
- 단, 상태 문구, 버튼 의미, 색상 토큰, 타이포 위계는 가능한 공통 기준을 유지한다.
- 따뜻하고 부드러운 인상을 유지하되, 운영 도구로서 빠르게 이해되는 구조를 우선한다.
- 카드로 모든 것을 해결하지 않는다. 카드가 없어도 되는 정보는 리스트 row, 텍스트 블록, 표 구조로 해결한다.
- 한 화면에서 메인 행동은 1개만 강조한다.
- 모바일 오너 앱 문법과 관리자 PC 콘솔 문법을 절대 섞지 않는다.

## 2. 화면군별 UI 기준

### A. 오너 모바일 앱

- 대상: `/owner`, 예약 조회, 고객관리, 설정, `/owner/billing`, `/owner/billing/success`
- 화면 목적:
  - 오너가 매일 반복적으로 조회, 수정, 저장, 응답하는 운영용 앱
- 레이아웃 기준:
  - `max-width: 430px`
  - 좌우 padding `16px`
  - 메인 탭 화면은 뒤로가기 없음
  - 하위 상세/편집 화면은 좌측 상단 뒤로가기
  - 닫기 버튼은 모달/바텀시트에서만 우측 상단
  - 메인 CTA는 하단 full-width 또는 섹션 하단
  - 바텀시트 중심
  - 카드 남용 금지
  - 리스트 row 중심
- 버튼 사용 기준:
  - 메인 저장/확정/결제/다음 행동만 primary
  - 보조 행동은 secondary 또는 text
  - 위험 행동은 별도 danger block
- 카드 사용 기준:
  - 요약 정보, 상태 요약, 묶음 섹션에만 사용
  - 긴 설명문, 긴 목록, 단순 문장에는 카드 사용 금지
- 텍스트 위계 기준:
  - 페이지 제목 > 섹션 제목 > 본문 > 보조 텍스트
  - 상태 배지는 보조 설명보다 강하고, 페이지 제목보다 약해야 한다
- 절대 피해야 할 패턴:
  - 카드 안 카드
  - 상단 우측 메인 CTA
  - 메인 탭 화면의 뒤로가기
  - 모바일 바텀시트 대신 임의의 인라인 확장 반복

### B. 소비자 예약 모바일 화면

- 대상: `/book/[shopId]`, `/book/[shopId]/info`, `/book/[shopId]/manage`, `/entry/[shopId]`
- 화면 목적:
  - 소비자가 빠르게 서비스 선택, 시간 선택, 예약 입력, 예약 조회를 완료하는 흐름
- 레이아웃 기준:
  - `max-width: 430px`
  - 단계형 플로우
  - 다음/예약하기 CTA는 하단 full-width
  - 이전 버튼 위치는 한 플로우 안에서 고정
  - 예약 중단/조회 종료 같은 레이어 상황에서만 우측 상단 닫기
  - 선택 카드, 요약 카드 외 카드 남용 금지
- 버튼 사용 기준:
  - 현재 단계의 메인 행동만 primary
  - 이전/취소/조회 종료는 secondary 또는 text
- 카드 사용 기준:
  - 서비스 선택, 예약 요약, 완료 확인용으로만 사용
  - 본문 텍스트 전달용 카드 남용 금지
- 텍스트 위계 기준:
  - 단계 제목과 진행 상태를 먼저 보여주고, 선택/입력 정보는 그 아래 배치
- 절대 피해야 할 패턴:
  - 단계별 CTA 위치가 매번 바뀌는 것
  - 이전 버튼 위치가 화면마다 달라지는 것
  - 예약 플로우 일반 페이지에 우측 상단 닫기 남용

### C. 관리자 PC 웹 콘솔

- 대상: `/admin`, `/admin/login`, `/admin/register`, `/admin/reset`, `/owner/admin`, `src/components/admin/owner-admin-screen.tsx`
- 화면 목적:
  - 관리자 계정이 목록 조회, 검색, 필터, 상세 확인, 운영 액션을 빠르게 수행하는 콘솔
- 레이아웃 기준:
  - `max-width: 1400px`
  - 상단 헤더 + 목록/테이블 + 상세 패널 구조
  - 검색/필터는 상단 또는 좌측 목록 상단
  - 상세 패널 닫기는 우측 상단
  - 모바일식 뒤로가기 버튼을 강제로 사용하지 않음
  - 모바일 full-width CTA를 반복 사용하지 않음
  - 위험 액션은 상세 패널 하단 danger block에 배치
  - 데이터 밀도는 모바일보다 높게 허용
- 버튼 사용 기준:
  - 목록 상단 액션, 상세 패널 액션, 위험 액션을 분리
  - 반복 full-width CTA 금지
- 카드 사용 기준:
  - 웹 콘솔에서는 카드보다 row, panel, table, toolbar 우선
  - 카드 사용 시 요약 정보 블록에 한정
- 텍스트 위계 기준:
  - 페이지 제목 > 패널 제목 > 섹션 제목 > row 메타 텍스트
- 절대 피해야 할 패턴:
  - 모바일 앱처럼 한 칸짜리 카드형 레이아웃 반복
  - full-width CTA 남용
  - 목록/상세 구조 없이 페이지를 계속 전환하는 방식

### D. 향후 오너 고객관리 PC 버전

- 대상: 아직 구현 전인 고객관리 PC 확장
- 화면 목적:
  - 오너가 더 많은 고객 데이터를 빠르게 조회, 필터, 편집, 일괄 관리하는 PC 전용 운영 화면
- 레이아웃 기준:
  - 모바일 고객관리 화면을 단순 확대하지 않음
  - 목록 + 상세 구조를 기본으로 함
  - 고객 검색/필터/정렬을 강화
  - 고객 row 또는 테이블 사용
  - 고객 상세는 우측 패널 또는 별도 상세 페이지
  - 모바일 바텀시트를 그대로 PC에 사용하지 않음
  - 모바일 full-width CTA를 그대로 PC에 사용하지 않음
  - 상태 배지, 고객명/연락처/반려동물 정보 위계는 모바일과 공통 유지
- 버튼 사용 기준:
  - 목록 상단에 고객 추가
  - 상세 패널 하단에 저장/삭제/정지 등 분리
- 카드 사용 기준:
  - 요약 블록 외 카드 남용 금지
  - 고객 목록은 카드보다 row/table 우선
- 텍스트 위계 기준:
  - 고객명 > 연락처 > 반려동물 > 메모/부가정보
- 절대 피해야 할 패턴:
  - 모바일 고객 리스트 카드를 그대로 넓히는 것
  - 모바일 바텀시트 구조를 PC 상세에 그대로 쓰는 것

### E. 인증/가입 화면

- 화면 목적:
  - 오너 계정 생성/로그인과 관리자 인증을 각각 다른 문법으로 명확히 처리
- 기준:
  - 오너 인증은 모바일 기준
  - 관리자 인증은 PC 관리자 콘솔 진입 기준
  - 두 화면의 목적이 다르므로 레이아웃 문법을 섞지 않음
  - 오너 인증은 입력 완료 중심
  - 관리자 인증은 접근 제어와 운영 도구 진입 중심
- 절대 피해야 할 패턴:
  - 관리자 로그인에 모바일 오너 앱 레이아웃을 그대로 쓰는 것
  - 오너 회원가입에 관리자 콘솔 패널 문법을 섞는 것

### F. 랜딩/법적 페이지

- 화면 목적:
  - 랜딩은 마케팅 전환
  - 법적/정책 페이지는 문서 전달
- 기준:
  - 랜딩은 마케팅 페이지 문법
  - 법적/정책 페이지는 텍스트 문서 문법
  - 앱 카드 문법 과다 적용 금지
- 절대 피해야 할 패턴:
  - 약관/정책 페이지를 앱 설정 화면처럼 구성하는 것
  - 랜딩 페이지를 오너 앱 카드 문법으로 구성하는 것

### G. 데모 페이지

- 화면 목적:
  - 실서비스 플로우를 설명하거나 시연하는 축약 화면
- 기준:
  - 실제 화면군 문법을 최대한 따라가되, 데모 전용 안내 블록만 별도로 둔다
  - 오너 데모는 오너 모바일 앱 문법
  - 소비자 데모는 소비자 예약 모바일 문법
- 절대 피해야 할 패턴:
  - 데모 전용 화면에 새로운 독자 문법을 만드는 것

## 3. 상단 내비게이션 / 뒤로가기 / 닫기 / CTA 위치 기준

### 오너 모바일 앱

- 메인 탭 화면은 뒤로가기 없음
- 하위 상세/편집 화면은 좌측 상단 뒤로가기
- 모달/바텀시트 닫기는 우측 상단
- 페이지 제목은 기본 좌측 정렬
- 상단 우측에 저장/결제/확정 같은 메인 CTA를 두지 않음
- 메인 CTA는 하단 full-width 또는 섹션 하단

### 소비자 예약 모바일

- 이전 버튼 위치는 한 플로우 안에서 고정
- 다음/예약하기 CTA는 하단 full-width
- 닫기 버튼은 예약 중단/조회 종료 같은 레이어 상황에서만 사용
- 단계 제목과 진행 상태 위치를 통일

### 관리자 PC 웹 콘솔

- 모바일식 뒤로가기 버튼 강제 사용 금지
- 상세 패널 닫기는 우측 상단
- 목록으로 돌아가는 기능은 breadcrumb, 목록 선택 상태, 패널 닫기로 처리
- 주요 액션은 상단 액션 영역 또는 패널 하단 고정 영역
- 위험 액션은 danger block에 분리

### 향후 고객관리 PC 버전

- 모바일 뒤로가기 대신 목록/상세 구조 사용
- 상세 패널 닫기 또는 breadcrumb 사용
- 모바일 하단 CTA를 그대로 사용하지 않음
- 고객 추가/저장/삭제 액션 위치를 PC 기준으로 별도 정의

### 금지 규칙

- 같은 화면군 안에서 뒤로가기 위치 좌우 혼용 금지
- 일반 페이지 이동에 우측 상단 닫기 버튼 사용 금지
- 모달/바텀시트 닫기에 좌측 뒤로가기 아이콘 사용 금지
- 모바일 오너 앱과 관리자 PC 콘솔의 CTA 위치 규칙 혼용 금지
- 모바일 고객관리 UI를 PC 버전에 단순 확대 적용 금지
- 관리자 PC 콘솔에 모바일 full-width CTA 반복 사용 금지

## 4. 타이포그래피 기준

| 항목 | font-size | font-weight | line-height | 사용 위치 |
| --- | --- | --- | --- | --- |
| Main Title | 28px | 700 | 36px | 서비스 종료, 결제 확인, 핵심 랜딩 타이틀 |
| Page Title | 24px | 700 | 32px | 화면 최상단 제목 |
| Section Title | 18px | 700 | 26px | 섹션 제목, 카드 그룹 제목 |
| Card Title | 16px | 700 | 24px | 카드 제목, 리스트 주요 제목 |
| Body | 14px | 400 | 22px | 본문, 설명문, 일반 입력 helper |
| Secondary Body | 13px | 400 | 20px | 보조 설명, 리스트 메타 |
| Caption | 12px | 500 | 18px | 캡션, 날짜, 보조 상태 |
| Error Text | 12px | 500 | 18px | 입력 에러, 경고 문구 |
| Button Text | 16px | 600 | 24px | primary/secondary 버튼 |
| Small Button Text | 14px | 600 | 20px | inline action, small button |
| Status Badge Text | 12px | 600 | 16px | 상태 배지 |

규칙:

- 한 화면에서 4단계 이상의 타이포 위계를 동시에 쓰지 않는다.
- `10px` 이하 텍스트 사용 금지
- 보조 정보가 CTA보다 강하게 보이지 않도록 한다.

## 5. 간격 시스템

### 기본 스케일

- `8px`
- `12px`
- `16px`
- `20px`
- `24px`
- `32px`
- `40px`
- `44px`
- `48px`

### 화면/섹션 기준

- 모바일 화면 좌우 padding: `16px`
- 관리자 웹 좌우 padding: `24px` ~ `32px`
- 섹션 간 간격: `24px`
- 카드 내부 padding: `16px` 또는 `20px`
- 카드 간 gap: `12px` 또는 `16px`
- 리스트 row 간격: `12px`
- 폼 필드 간격: `16px`
- 버튼 상단/하단 간격: `16px` ~ `24px`

규칙:

- 임의 spacing 금지
- 같은 레벨의 반복 컴포넌트는 같은 gap을 사용

## 15. Floating Label Fieldset 기준

### 적용 대상

- 모바일 설정 화면의 `fieldset + legend` 구조
- 입력 묶음을 카드처럼 감싸되, 별도 상단 라벨 줄을 만들지 않는 경우
- 예: `SettingsFieldCard`

### 위치 기준

- 라벨 시작선은 `ml-0.5` 기준으로 맞춘다.
- 실제 픽셀 기준 약 `2px`이다.
- 목적은 라벨이 왼쪽으로 타이트하게 붙어 보이게 하는 것이다.
- `ml-2` 이상으로 라벨을 오른쪽으로 밀지 않는다.

### 라벨 박스 기준

- 좌우 내부 여백은 `px-1.5`
- 실제 픽셀 기준 약 `6px`
- 라벨은 테두리 위에 얹히되, 배경색은 부모 surface와 자연스럽게 이어지게 처리한다.

### 타이포 기준

- font-size: `14px`
- font-weight: `500`
- tracking: `-0.01em`
- color: `var(--muted)`

### 사용 규칙

- 같은 화면군 안에서는 floating label 시작선을 제각각 다르게 두지 않는다.
- 텍스트가 긴 라벨이라도 시작선은 동일하게 유지한다.
- 입력 콘텐츠 시작선보다 라벨이 과하게 오른쪽으로 밀려 보이면 위치 기준을 다시 맞춘다.

### 금지

- fieldset마다 라벨 시작선을 임의로 조정하는 것
- 라벨 padding을 과하게 키워서 오른쪽으로 밀려 보이게 만드는 것
- floating label 구조와 일반 상단 label 구조를 같은 묶음 안에서 섞는 것

## 16. 고객 상세 카드 본문 정렬 기준

### 적용 대상

- 오너 모바일 앱의 고객 상세 화면 카드
- 예: `기본 정보`, `고객 메모`, `알림톡 설정`, `미용 기록`, `알림톡 발송 내역`

### 목적

- 카드마다 본문 텍스트 시작선이 달라 보이지 않게 한다.
- `알림톡 수신`, `재방문 안내`, `미용 기록`처럼 같은 계열 카드의 본문이 같은 축에서 시작하게 한다.
- 카드 안 텍스트가 아래로 처지거나, 카드마다 위아래 여백이 달라 보이는 문제를 막는다.

### 안쪽 래퍼 기준

- 카드 본문 안쪽 래퍼는 `px-3.5 py-2`를 기본으로 한다.
- 본문 래퍼의 세로 위치 보정은 `relative -top-0.5`까지 허용한다.
- 새로운 카드도 같은 계열이면 먼저 이 기준으로 맞춘다.

### 본문 시작선 기준

- 본문 텍스트 시작선은 `알림톡 설정` 카드의 `알림톡 수신` 시작선을 기준으로 맞춘다.
- `미용 기록` 카드의
  - 서비스명
  - 반려동물명
  - 날짜
  - 스타일 메모
  - 상세 메모
  는 모두 이 시작선과 같은 축에서 시작해야 한다.
- 카드마다 본문에 `pl-*`, `ml-*`, `px-0.5` 같은 개별 보정값을 추가하지 않는다.

### 세로 균형 기준

- 카드 안 본문은 시각적으로 가운데보다 약간 위에 있도록 맞춘다.
- 카드 위쪽 여백과 아래쪽 여백은 카드 간 동일한 규칙을 사용한다.
- 같은 상세 카드군에서 어떤 카드만 위가 넓고 아래가 좁아 보이면 래퍼 기준을 다시 맞춘다.

### 금지

- 카드마다 본문 시작선을 임의로 따로 미세조정하는 것
- 특정 카드에만 추가 left padding을 넣어 기준선이 밀리게 하는 것
- 같은 카드군인데 어떤 카드는 `px-3`, 어떤 카드는 `px-3.5`, 어떤 카드는 `pl-2`처럼 섞어 쓰는 것
- 카드 안 본문 텍스트를 개별적으로 아래로 내리거나 위로 올려 기준축이 깨지게 만드는 것
