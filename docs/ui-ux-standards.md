# UI/UX Standards

이 문서는 `pawcare` 프로젝트의 화면 기준을 정리한 공통 규칙 문서다.
현재 프로젝트에 바로 적용할 수 있어야 하고, 이후 다른 모바일 중심 SaaS 프로젝트에도 재사용할 수 있도록 작성한다.

## 1. 문서 목적

- 화면마다 다른 톤, 간격, 카드 크기, 버튼 높이, 문구 밀도를 통일한다.
- 새 화면을 만들 때 즉흥적으로 스타일을 쌓지 않고, 같은 기준 안에서 변형한다.
- 기획, 디자인, 개발이 같은 언어로 이야기할 수 있게 숫자와 원칙을 먼저 고정한다.

## 2. 제품 기본 전제

- Korean-first UX를 기본으로 한다.
- 모바일 우선, 최대 너비는 `430px`로 고정한다.
- 과한 장식보다 입력 완성도, 정보 위계, 간격 품질로 완성도를 만든다.
- 오너용 SaaS는 신뢰감과 정돈감이 먼저고, 고객 예약 화면은 접근성과 쉬운 사용성이 먼저다.
- 마케팅, 인증, 오너 앱, 고객 예약, 관리자 화면은 같은 시스템 안에서 역할이 다른 화면군으로 본다.

## 3. 화면군 정의

### 3.1 마케팅 / 랜딩

- 목적: 설득, 기능 설명, 진입 CTA
- 특징: 감정 톤이 가장 강해도 됨
- 허용: 큰 타이틀, 강조 배지, 히어로 블록
- 금지: 앱 내부 정보 구조를 그대로 가져오는 것

### 3.2 인증

- 목적: 로그인, 회원가입, 기본정보 입력, 계정 복구
- 특징: 가장 단정하고 미니멀해야 함
- 허용: 짧은 설명, 적은 수의 입력, 명확한 CTA
- 금지: 과한 카드 장식, 설명 과다, 한 화면 다중 CTA 난립

### 3.3 오너 앱

- 목적: 운영, 예약 처리, 고객 관리, 설정
- 특징: 정보 밀도가 가장 높음
- 허용: 카드, 탭, 상태 뱃지, 섹션 구분
- 금지: 마케팅성 카피, 과한 색 강조, 큰 빈 여백

### 3.4 고객 예약

- 목적: 예약 신청, 재방문 예약, 예약 확인/변경/취소
- 특징: 쉬운 흐름, 입력 부담 최소화
- 허용: 단계 표시, 친절한 설명, 간단한 상태 피드백
- 금지: 관리자용 용어, 복잡한 옵션 노출

### 3.5 관리자

- 목적: 오너 계정 제어, 기간 조정, 로그 확인
- 특징: 기능 우선, 밀도 높음
- 허용: 표, 상태값, 빠른 액션
- 금지: 마케팅형 큰 블록, 과한 장식

## 4. 레이아웃 기준

### 4.1 앱 기본 프레임

- 기본 컨테이너: `mx-auto w-full max-w-[430px]`
- 기본 좌우 패딩:
  - 모바일: `px-4` 또는 `px-5`
  - 인증/입력 중심: `px-5` 권장
- 기본 하단 여백:
  - 페이지 종료 전: `pb-8` 이상
  - CTA 고정 시: CTA 높이 + `pb-6`

### 4.2 섹션 간격

- 큰 섹션 간격: `24px`
- 일반 블록 간격: `16px`
- 조밀한 정보 행 간격: `12px`
- 같은 카드 안의 라벨/값 간격: `4px`~`6px`

### 4.3 카드 사용 원칙

- 카드가 필요할 때만 사용한다.
- 카드가 없어도 정보 구분이 가능한 화면은 카드보다 여백과 구분선으로 해결한다.
- 인증 화면은 가능한 한 카드 바깥 구조를 기본으로 한다.
- 카드가 많아질수록:
  - 높이 축소
  - 내부 패딩 축소
  - 그림자 축소
  - 정보 위계 강화
  를 같이 해야 한다.

## 5. 타이포그래피 기준

### 5.1 타입 스케일

- 페이지 라벨: `12px` / `font-medium`
- 섹션 라벨: `13px` / `font-medium`
- 본문: `14px`
- 보조 본문: `13px`
- 필드 라벨: `12px`
- 카드 제목: `15px`~`17px`
- 페이지 제목:
  - 인증/폼: `28px`~`32px`
  - 앱 내부: `24px`~`28px`
- 숫자 강조(가격/집계): `24px`~`32px`

### 5.2 줄간격

- 본문 기본: `leading-6`
- 짧은 보조 문구: `leading-5`
- 카드 타이틀/숫자: `leading-[1.1]` 또는 `leading-tight`

### 5.3 타이포 원칙

- 한 화면에 굵기 레벨은 많아도 3단계까지만 사용한다.
- 제목과 숫자는 크기로 강조하고, 보조 정보는 색과 줄간격으로 눌러준다.
- 회색 정보는 읽히되 시선 뺏지 않는 수준까지 낮춘다.

## 6. 크기 기준

### 6.1 입력창

- 기본 높이: `48px`~`52px`
- 텍스트 영역: `min-h-[96px]` 이상
- 라벨은 입력창 내부가 아니라 위나 필드 상단에 놓는 것을 우선한다.

### 6.2 버튼

- 기본 CTA 높이: `52px`
- 작은 버튼: `40px`~`44px`
- pill형 버튼은 특별한 목적이 아닐 때 남발하지 않는다.
- 1차 CTA는 한 화면에 1개를 기본으로 한다.

### 6.3 카드

- 기본 라운드: `16px`~`20px`
- 강조 카드: 최대 `24px`
- `28px` 이상 라운드는 마케팅/히어로가 아니면 지양한다.

## 7. 색상 역할 기준

색상은 값보다 역할로 관리한다.

- `--text`: 기본 텍스트
- `--muted`: 보조 텍스트
- `--background`: 페이지 배경
- `--surface`: 카드/입력 배경
- `--border`: 기본 경계선
- `--accent`: 핵심 CTA / 주요 상태
- `--accent-soft`: 강조의 약한 배경
- `--danger`: 오류 / 삭제 / 경고
- `--success`: 완료 / 정상

원칙:

- 한 화면에서 주 강조색은 1개만 사용한다.
- 경고/실패 색은 메시지와 파괴적 액션에만 쓴다.
- 보조 정보까지 메인 컬러를 쓰지 않는다.

## 8. 공통 컴포넌트 규칙

### 8.1 버튼

- `Primary`
  - 가장 중요한 CTA
  - 페이지당 1개 원칙
- `Secondary`
  - 보조 CTA
  - 보더 + 흰 배경
- `Ghost`
  - 텍스트 기반
  - 파괴적 액션 또는 보조 이동에 사용
- `Danger`
  - 삭제, 취소 확정, 정지

금지:

- 비슷한 역할의 버튼이 화면마다 높이와 radius가 달라지는 것
- 버튼마다 다른 그림자 규칙을 쓰는 것

### 8.2 입력창

- 라벨은 짧게 쓴다.
- placeholder는 설명이 아니라 예시만 넣는다.
- 검증 에러는 입력창 바로 아래 또는 관련 블록 바로 위에 둔다.
- 에러 문구는 빨간색 + 짧은 한 문장으로 고정한다.

### 8.3 카드

- 카드 내부는 최소 2개 이상의 정보 그룹이 있을 때만 사용한다.
- 카드 안에 카드가 또 들어가는 구조는 되도록 피한다.
- 카드 안 텍스트가 너무 커지면 카드가 둔해 보이므로, 정보 밀도가 높아질수록 박스를 줄이고 텍스트 위계를 조정한다.

### 8.4 탭

- 탭 이름은 2~6글자 내로 짧게 유지한다.
- active 탭만 색을 쓰고, inactive 탭은 최대한 조용하게 둔다.
- 탭이 많으면 pill보다 underline 또는 segmented control이 낫다.

### 8.5 상태 뱃지

- 뱃지는 정보 요약용이다.
- 크기보다 대비로 읽히게 한다.
- 작은 텍스트 + 낮은 높이를 기본으로 한다.

### 8.6 빈 상태

- 빈 상태는 큰 일러스트보다 짧은 설명 + 행동 유도 1개를 우선한다.
- 관리자/오너 화면은 특히 과한 빈 상태를 피한다.

## 9. 인증 화면 규칙

- 헤더:
  - 뒤로가기
  - 작은 라벨
  - 제목
  - 짧은 설명
- 필드는 1열 정렬
- 약관은 필요 시점에만 노출
- 소셜 로그인은 입력 폼 아래에 두고, 버튼 규격은 통일한다
- 로그인/회원가입/기본정보 입력은 서로 톤이 같아야 한다

인증 화면 금지:

- 과한 카드 중첩
- 설명 문구 3줄 초과
- 공급자(provider) 문구를 매 화면 상단에서 크게 반복

## 10. 오너 앱 규칙

- 홈 화면은 정보 우선 순위가 바로 보여야 한다
- 예약 상태, 고객 정보, 설정은 서로 다른 밀도의 정보군으로 분리한다
- 상단 액션은 1~2개로 제한한다
- 고객 상세는:
  - 기본정보
  - 아기 정보
  - 미용 기록
  - 알림톡 발송내역
  구조를 유지
- 관리자용 제어는 일반 오너 화면과 분리한다

## 11. 고객 예약 화면 규칙

- 첫 방문 / 재방문 / 예약 확인은 서로 다른 목적이므로 블록을 섞지 않는다
- 한 단계에서 묻는 정보 수는 최소화한다
- 피드백 문구는 즉시 보이게 한다
- 보안/검증이 필요한 경우도 입력 부담이 과도하지 않게 조정한다

## 12. 문구 원칙

- 짧고 실무적인 한국어를 쓴다.
- 마케팅 페이지 외에는 과장된 표현을 줄인다.
- 설명은 가능한 한 한 문장으로 끝낸다.
- 버튼 문구는 동사 중심으로 쓴다.

예:

- 좋은 예
  - `기본 정보를 입력해 주세요`
  - `무료체험 시작하기`
  - `서비스 30일 연장`
- 나쁜 예
  - `지금 바로 경험해보세요!`
  - `손쉽고 빠르게 편리하게 시작할 수 있어요`

## 13. 네비게이션 원칙

- 마케팅(`/`)과 실사용 앱(`/owner`)은 분리한다.
- 데모(`/demo/...`)는 실사용 라우트와 섞지 않는다.
- 인증 완료 후 앱 진입은 `replace`를 우선 사용한다.
- 이미 로그인된 사용자가 마케팅/인증으로 들어오면 앱으로 돌려보내는 것을 기본으로 한다.

## 14. 구현 원칙

- Tailwind 클래스는 즉흥적으로 길어지기 전에 역할별 상수/헬퍼로 묶는다.
- 새 화면을 만들기 전 먼저:
  - 이 화면이 속한 화면군
  - CTA 1순위
  - 정보 밀도
  를 정한다.
- 공통 컴포넌트 우선순위:
  1. 버튼
  2. 입력창
  3. 섹션 헤더
  4. 카드
  5. 바텀시트/모달
  6. 탭
  7. 상태 뱃지

## 15. QA 체크리스트

새 화면 또는 큰 수정 후 아래를 확인한다.

- 한눈에 이 화면 목적이 보이는가
- CTA가 하나로 명확한가
- 같은 역할의 버튼이 다른 화면과 규격이 같은가
- 입력창 높이와 라벨 위치가 통일되어 있는가
- 카드가 필요 이상으로 크지 않은가
- 정보 사이가 너무 퍼져 보이지 않는가
- 오류 문구 위치가 입력 흐름을 방해하지 않는가
- 모바일 `430px` 기준에서 답답하거나 지나치게 붕 뜨지 않는가
- 뒤로가기/replace 흐름이 사용자 의도와 맞는가

## 16. 현재 프로젝트 우선 정리 대상

현재 코드 기준으로 우선순위가 높은 정리 대상은 아래다.

1. 인증 화면군
   - 로그인
   - 회원가입
   - 소셜 기본정보 입력
2. 오너 홈 / 설정 / 플랜 카드
3. 고객 예약 / 예약 확인 / 재방문 예약
4. 공통 푸터 / 바텀시트 / 에러 상태
5. 오설페 마케팅 블록

## 16.1 기준 화면 고정

이 프로젝트는 이미 완성도가 높은 화면을 기준 화면으로 고정하고, 다른 화면이 그 기준을 따라가도록 정리한다.

### 인증 기준 화면

- 기준 화면: [D:\pawcare\src\components\auth\login-form.tsx](D:/pawcare/src/components/auth/login-form.tsx)
- 이유:
  - 정보 위계가 명확함
  - CTA 우선순위가 뚜렷함
  - 버튼/입력/보조 링크 간 균형이 좋음
  - 과한 장식 없이 단정한 SaaS 톤을 유지함

앞으로 아래 화면은 로그인 화면의 결을 따라간다.

- 회원가입
- 소셜 기본정보 입력
- 아이디 찾기
- 비밀번호 재설정

정확히 맞춰야 할 요소:

- 상단 타이틀/설명 톤
- 입력창 높이와 보더 강도
- 메인 CTA 비중
- 보조 링크 밀도
- 소셜 버튼 간 정렬과 세로 간격

허용되는 차이:

- 단계 표시 유무
- 입력 필드 수
- 보조 설명 문구의 길이

허용되지 않는 차이:

- 버튼 규격이 다른 것
- 입력창 높이와 radius가 크게 다른 것
- 같은 인증 흐름인데 다른 앱처럼 보이는 것

## 17. 적용 방식 제안

실행 순서는 이렇게 간다.

1. 이 문서를 기준 문서로 고정
2. 공통 버튼 / 입력 / 카드 규격 추출
3. 인증 화면군 일괄 정리
4. 오너 화면군 일괄 정리
5. 고객 예약 화면군 정리
6. 마케팅 화면 마지막 정리

## 18. 한 줄 기준

이 프로젝트의 화면은

`작고 명확한 위계, 얇은 구조, 적은 설명, 강한 CTA 1개`

를 기본 규칙으로 삼는다.
